# Defense Q&A

## 1. Яка мета проєкту?

EventFlow — консольна система управління подіями, яка дозволяє створювати події, реєструвати учасників, скасовувати реєстрації та зберігати дані між запусками програми.

---

## 2. Яку архітектуру використано?

Використано багатошарову архітектуру:

```text
Console
↓
Application
↓
Domain
↓
Infrastructure
```

Кожен шар має власну відповідальність.

---

## 3. Які принципи SOLID реалізовано?

### SRP

Кожен сервіс має одну відповідальність.

### OCP

Систему можна розширювати новими реалізаціями без зміни існуючого коду.

### LSP

Реалізації репозиторіїв можна замінювати через інтерфейси.

### ISP

Використовуються вузькі інтерфейси.

### DIP

Сервіси залежать від абстракцій, а не від конкретних реалізацій.

---

## 4. Які патерни використано?

### Repository Pattern

Для доступу до даних через абстракції.

### Factory Pattern

Для створення об'єктів подій.

### Result Pattern

Для контрольованої обробки помилок.

---

## 5. Чому використано JSON замість бази даних?

Для навчального проєкту JSON є простим способом реалізувати persistence без ускладнення архітектури.

---

## 6. Як реалізовано persistence?

Використовуються:

- IDataStore;
- JsonEventStore;
- EventPersistenceService;
- EventMapper.

Дані серіалізуються у JSON і відновлюються під час запуску програми.

---

## 7. Як перевіряється коректність реєстрації?

Система не дозволяє:

- перевищувати місткість події;
- дублювати реєстрації;
- реєструватися на завершені події.

---

## 8. Для чого потрібен EventQueryService?

Для виконання LINQ-запитів:

- фільтрації;
- сортування;
- пошуку;
- побудови статистики.

---

## 9. Які колекції використовуються?

- List<T>
- IReadOnlyCollection<T>
- Dictionary<TKey, TValue>

---

## 10. Які тести реалізовано?

### Unit Tests

Перевіряють бізнес-логіку.

### Integration Tests

Перевіряють взаємодію між компонентами та persistence.

### Fault Handling Tests

Перевіряють негативні сценарії.

---

## 11. Як реалізовано Quality Gate?

GitHub Actions автоматично виконує:

- restore;
- build;
- test;
- coverage.

При помилці тестів збірка завершується невдачею.

---

## 12. Які ризики залишаються?

- Відсутність бази даних.
- Відсутність логування.
- Простий консольний інтерфейс.
- Відсутність Dependency Injection Container.

---

## 13. Що було найскладнішим?

Найскладнішою частиною було коректне відновлення агрегату Event після десеріалізації JSON та побудова тестів для persistence-шару.

---

## 14. Як забезпечується якість коду?

- SOLID;
- Unit Tests;
- Integration Tests;
- Coverage;
- CI Pipeline;
- Рефакторинг.
