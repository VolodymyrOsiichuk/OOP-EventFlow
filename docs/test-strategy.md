# Test Strategy

## Мета

Забезпечити стабільність проєкту EventFlow шляхом автоматизованого тестування критичних бізнес-сценаріїв, persistence-шару та обробки помилок.

---

## Критичні сценарії

### Створення події

Ризики:

- некоректна дата;
- від'ємна або нульова місткість;
- порожня назва.

### Реєстрація учасника

Ризики:

- перевищення ліміту;
- дублювання реєстрації;
- реєстрація на завершену подію.

### Скасування реєстрації

Ризики:

- учасник відсутній;
- повторне скасування.

### Persistence

Ризики:

- відсутній файл;
- пошкоджений JSON;
- втрата даних після Save/Load;
- дублювання даних після повторного завантаження.

---

## Найскладніші частини для тестування

### JSON Persistence

Причина:

- робота з файловою системою;
- необхідність ізоляції тестових файлів.

### EventMapper

Причина:

- відновлення агрегату Event;
- коректне відновлення Registration та Participant.

### EventPersistenceService

Причина:

- взаємодія кількох компонентів одночасно.

---

## Де використовуються mocks

- IDataStore
- IEventRepository

---

## Де потрібні інтеграційні тести

- SaveAsync()
- LoadAsync()
- Повний цикл створення події
- Повторне завантаження даних

---

## Негативні сценарії

- реєстрація дубліката;
- перевищення місткості;
- реєстрація на завершену подію;
- скасування неіснуючої реєстрації;
- відсутній JSON файл;
- пошкоджений JSON файл;
- повторне завантаження тих самих даних.

---

## Test Pyramid

### Unit Tests

Основна частина тестів.

Покривають:

- Event;
- Participant;
- EventService;
- EventQueryService.

### Integration Tests

Перевіряють:

- Persistence;
- Save/Load;
- взаємодію між шарами.

### Manual Tests

Перевірка консольного меню та UX.
